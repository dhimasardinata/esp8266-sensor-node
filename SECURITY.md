# Security Policy

Kami menganggap keamanan proyek Greenhouse Node Firmware ini sebagai prioritas utama. Kami menghargai upaya para peneliti keamanan dan komunitas untuk membantu kami menjaga keamanan proyek ini.

## Supported Versions

Karena ini adalah proyek firmware untuk perangkat keras fisik, kebijakan kami adalah hanya mendukung **versi stabil terbaru** yang tersedia di branch `main`. Perangkat dirancang untuk diperbarui secara otomatis atau manual melalui fitur Over-the-Air (OTA).

| Version | Supported |
| :--- | :--- |
| **Latest Stable (`main` branch)** | :white_check_mark: |
| Older Versions (< latest) | :x: |

Pengguna sangat dianjurkan untuk selalu memastikan perangkat mereka menjalankan firmware terbaru untuk menerima semua perbaikan keamanan dan fungsionalitas.

## Reporting a Vulnerability

Kami berkomitmen untuk bekerja sama dengan siapa pun yang melaporkan masalah keamanan. **Harap JANGAN melaporkan kerentanan keamanan melalui Isu publik di GitHub.**

Untuk melaporkan kerentanan, silakan gunakan salah satu dari dua metode berikut, diurutkan berdasarkan preferensi:

### 1. GitHub Private Vulnerability Reporting (Metode Pilihan)

Ini adalah cara paling aman dan efisien untuk melaporkan kerentanan. Fitur ini memungkinkan kita berdiskusi dan berkolaborasi secara pribadi.

1.  Navigasikan ke tab **"Security"** di repository GitHub ini.
2.  Klik tombol **"Report a vulnerability"**.
3.  Isi formulir dengan detail kerentanan.

### 2. Laporan Melalui Email

Jika Anda tidak dapat menggunakan metode di atas, Anda dapat mengirim email laporan Anda ke:

**dhimasardinatapp@gmail.com**

---

### Detail yang Perlu Disertakan dalam Laporan Anda

Untuk membantu kami memahami dan menyelesaikan masalah secepat mungkin, mohon sertakan informasi berikut dalam laporan Anda (jika memungkinkan):

*   **Tipe Kerentanan:** (Contoh: Buffer Overflow, Authentication Bypass, Remote Code Execution, dll.)
*   **Deskripsi Detail:** Penjelasan yang jelas tentang kerentanan dan potensi dampaknya.
*   **Langkah-langkah untuk Mereproduksi:** Langkah-langkah yang tepat untuk mereplikasi kerentanan pada versi firmware terbaru.
*   **Proof-of-Concept (PoC):** Skrip atau kode yang dapat digunakan untuk mendemonstrasikan eksploitasi.
*   **Versi Firmware yang Terdampak:** Versi spesifik dari firmware yang Anda uji.

### Proses dan Komitmen Kami

Setelah Anda melaporkan kerentanan, berikut adalah apa yang dapat Anda harapkan dari kami:

1.  **Konfirmasi Penerimaan:** Kami akan memberikan konfirmasi bahwa kami telah menerima laporan Anda dalam waktu **3-5 hari kerja**.
2.  **Investigasi & Validasi:** Kami akan menyelidiki laporan Anda dan berupaya memvalidasi kerentanan. Kami mungkin akan menghubungi Anda jika kami memerlukan informasi lebih lanjut.
3.  **Rencana Perbaikan:** Setelah kerentanan divalidasi, kami akan mulai mengerjakan perbaikan.
4.  **Rilis & Pengungkapan:** Setelah perbaikan siap, kami akan merilis versi firmware baru yang telah ditambal. Kami akan mengkreditkan Anda atas penemuan Anda (kecuali jika Anda ingin tetap anonim) dalam catatan rilis. Kami akan mengoordinasikan pengungkapan publik dengan Anda.

Kami berterima kasih atas kontribusi Anda dalam menjaga keamanan proyek ini.
